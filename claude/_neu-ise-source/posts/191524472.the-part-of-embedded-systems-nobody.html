There is a peculiar kind of trust embedded in the word firmware. Firm. As if the software closest to the machine has earned a solidity that other code has not. We talk about hardware and software as if they occupy opposite ends of a spectrum, and firmware is the hinge—the negotiated zone where human intention meets physical law. When the code runs at that level, it does not fail gracefully. It fails finally. The airplane does not warn you. The medical device does not suggest a restart. The industrial controller does not apologize.

I find myself returning to this when I encounter a technical document that turns out to be, quietly, a document about accountability. “Engineering Safe Interoperability: Architectural Strategies for Integrating C and Rust in ARM Cortex-M Firmware” is, on its surface, a paper about calling conventions, memory alignment, and linker scripts. But underneath the architecture diagrams and the code snippets, it is asking something older: What does it mean to inherit a world someone else built, and is it possible to make it safer without destroying what it is?

The answer the paper proposes is neither revolution nor capitulation. It is something more interesting and more honest—a hybrid, a negotiation, a coexistence that acknowledges the reality of legacy systems while insisting that safety is not a luxury. That insistence is the paper’s moral argument, dressed in technical language.

The Inheritance Problem

C is fifty years old. That is not a flaw. That is a provenance. The language carries within it the fingerprints of every embedded system ever built—every hardware abstraction layer from every vendor, every network stack, every driver for every peripheral that a microcontroller has ever been asked to control. You do not discard fifty years of civilization because newer people have figured out what some of the problems were. You negotiate with it.

The paper understands this. It does not argue for a Rust utopia in which C is banished and all firmware is rewritten from scratch. It argues for something harder to execute and more honest about how institutions actually change: wrap it. Build safe abstractions atop the existing C, enforce Rust’s guarantees at the new edges, and let the legacy code continue to do what it was written to do while ensuring it can no longer reach into the new code and corrupt it.

This is the extern "C" block. This is #[no_mangle]. This is the entire apparatus of the Foreign Function Interface—a set of conventions that allows two languages to coexist in the same binary by agreeing, formally and precisely, on what they owe each other. The C code promises to behave according to the ARM EABI. The Rust code promises not to mangle the names it needs to find. They shake hands across the boundary.

What the paper is describing, at its core, is a theory of institutional change disguised as a theory of software architecture. You do not raze the building. You reinforce the foundation.

What Memory Actually Is

Memory safety is not an abstract virtue. It is the difference between a program that does what it was told and a program that does something else—something that no one intended, something that an attacker can exploit or a bug can trigger or a race condition can corrupt.

Buffer overflows. Use-after-free. Data races. These are not theoretical failure modes. They are the documented mechanism behind an astonishing percentage of critical security vulnerabilities. The NSA, an institution not historically known for advocacy, has in recent years recommended the transition away from C and C++ for new development precisely because of these vulnerabilities. The problem is not obscure. It is structural.

C’s model of memory is trust extended without verification. You, the programmer, are responsible for every allocation and deallocation. You are responsible for checking that the pointer is not null before you dereference it. You are responsible for ensuring that the buffer is large enough. You are responsible for knowing when you have given something up. The language will not remind you. It trusts you, completely, and the cost of that trust, measured across decades of deployed software, has been catastrophic.

Rust’s model is different in kind, not degree. It does not make you more careful. It makes carelessness impossible to compile. The ownership system, the borrow checker, the lifetime annotations—these are not ornamental. They are the compiler refusing to produce a binary that contains a known class of memory error. The safety is not runtime protection. It is compile-time proof.

The paper documents what happens at the boundary between these two philosophies. Every FFI call is a moment of negotiation. Rust code that calls into C must be wrapped in unsafe—an explicit, syntactic acknowledgment that here, the guarantees end. The programmer is taking responsibility back from the compiler. The paper’s recommended pattern is to minimize these moments, wrap them tightly in safe Rust functions that validate inputs and enforce invariants, and ensure that the unsafety cannot leak into the rest of the application.

This is not merely good engineering. Every recommendation it makes forces the developer to name what is happening—to document what is trusted and what is verified, to create boundaries that make violations visible. The unsafe keyword refuses to let you pretend that calling a C function is the same as calling a Rust function. The asymmetry is honest. That honesty is a moral claim.

The Architecture of Accumulated Decisions

A linker script is a contract with the physics of the device—and merging two of them is the engineering equivalent of merging two constitutions. It tells the toolchain where everything lives: the executable code in FLASH, the mutable data in RAM, the interrupt vector table at the precise memory address the hardware expects it to be. In a Cortex-M microcontroller, the linker script is not a suggestion. It is a map. Get it wrong, and the program does not misbehave in an interesting way. It simply does not run.

What the paper describes—the need to reconcile linker scripts when merging a C project into Rust—requires finding a unified document that honors both sets of claims, or adjudicating where they conflict. Both documents were written by people who assumed they were the only ones writing. Both make claims about the same territory.

“Orphan sections,” the paper notes in passing, are what happens when the linker cannot find a home for a segment of code. The linker places them somewhere—it has to put them somewhere—but where they land may be arbitrary. In a system where memory regions have specific permissions, an arbitrary placement can produce a hard fault the moment the processor tries to execute code it was told to treat as data.

The firmware crashes. The device freezes. The consequences are not proportional to the size of the error. They are proportional to what the system was doing.

In most software, incorrect code produces a wrong answer. In embedded firmware, incorrect code kills people. Not always. Not inevitably. But the failure modes of embedded systems are not buffered by operating systems or hardware abstractions or network timeouts. When the bare-metal code fails, it fails all the way down.

The paper knows this. It is why the rigor is not intellectual vanity. It is the acknowledgment that someone, somewhere, is depending on this working.

The Trampoline and What It Means

There is a pattern in the paper I keep returning to. It is called the trampoline, and it solves a specific problem: C APIs use function pointers for callbacks, and Rust closures—which capture their environment and have unknown size—cannot be passed directly as function pointers.

The solution is elegant. You create a static C-compatible function that acts as an intermediary. C calls the trampoline. The trampoline casts the context pointer back to a Rust closure and calls it. C’s requirement is satisfied. Rust’s ergonomics are preserved. The two systems talk to each other through a mediator that speaks both languages.

I find in this pattern an allegory for what the paper is actually doing. The entire project of hybrid Rust/C firmware is a trampoline—a mediating structure that allows two incompatible systems to interoperate without either one having to become the other. You do not force the C HAL to become memory-safe. You do not force Rust to abandon its ownership model. You build a structure in between that honors both sets of constraints, and you are precise and explicit about where the translation happens.

The paper is careful to note that the trampoline requires careful lifetime management. If the C library outlives the closure, the trampoline will dereference a dangling pointer. The safety is not automatic. It has to be designed in. The pattern solves the syntactic problem; the architect still has to solve the semantic one.

This is the condition of all mediation. The structure can facilitate. It cannot substitute for judgment.

What the Paper Asks of Us

The cost of memory unsafety is not distributed evenly. The embedded Rust community understands this. This paper articulates it.

The failures concentrate in the systems where people cannot afford for them to happen. The medical devices. The industrial controllers. The automotive systems. The satellites. These are systems whose failures do not issue error reports. They issue consequences.

The incremental adoption strategy the paper recommends is not timidity. Start at the edges. Write the new features in Rust. Keep the existing C drivers. Establish the safety guarantee in the new code, and let that guarantee expand as the team gains confidence and the system earns trust. This is how institutions actually change when they are serious about it—not through proclamation, but through the patient accumulation of better decisions.

Rust does not solve the problem of bad architecture. It does not prevent misplaced linker sections or incorrect timer configurations or wrong SPI clock polarity. What it eliminates is an entire class of failure mode that has no business existing in software written by professionals in the twenty-first century. The buffer overflow. The use-after-free. The data race between an interrupt handler and the main loop.

The compiler can check these things. We decided, for fifty years, that checking them was the programmer’s job. We were wrong. Not because programmers are incompetent—we were wrong because no programmer, no matter how skilled, should be required to manually verify every pointer dereference in a quarter-million-line codebase. The machine can do it. The machine should do it.

The paper’s final recommendation is this: leverage the existing ecosystem rather than rebuilding from scratch. The embedded Rust community has done the hard work of building the tools. The question for every firmware team is whether they are willing to do the work of learning to use them.

That question is not technical. It is a question about what you owe to the person who depends on the device you are building.

The answer has always been the same. Everything.
