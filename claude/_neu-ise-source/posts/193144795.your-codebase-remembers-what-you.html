I came back to a project after ten days away. The code was exactly where I left it. The logic was clean, the structure was solid — and I had absolutely no idea why I’d made one of the most important architectural decisions in the whole thing.

It wasn’t in a comment. It wasn’t in a commit message. It existed somewhere in the 47-tab browser session I’d closed on a Tuesday night when I thought I was done for the day.

I spent two hours reverse-engineering my own thinking. Reading old code like a detective. Checking commit messages that said things like “fix issue” and “refactor.” Skimming a Notion doc I’d half-written two weeks earlier and abandoned mid-sentence.

I eventually figured it out. But that two hours wasn’t engineering. It was archaeology.

That moment is what I now call context decay. And every builder I know has felt it.

The code survives. The reasoning doesn’t.

Across three projects I built over the past year — a local AI pipeline, a financial data system, and a multimodal retrieval tool — I kept running into the same problem in different forms.

In one project, I wrote numbered SQL migration files not because the schema required it, but because future-me needed to reconstruct why the schema looked the way it did. In another, I wrote session handoff documents at the end of every work session — essentially letters to myself — so I could resume without losing a week re-reading code. In a third, I planned an entire DESIGN.md and an architecture-decisions.md file as first-class project artifacts, before I’d written a single function.

None of that was solving an engineering problem. It was manually doing what a tool should have been doing for me.

The code was versioned. The decisions weren’t.

What we call documentation is really just archaeology.

We write things down after the decision, imperfectly, from memory. We capture what changed — a diff, a migration, a refactor — but rarely why. The constraint that forced the decision. The alternative that was seriously considered and rejected. The tradeoff that felt obvious in the moment and is completely invisible six weeks later.

When we return to a codebase — whether it’s ten days or ten months later — we’re digging through commits and comments trying to reconstruct intent from artifacts. That’s not knowledge management. That’s guesswork with version control.

So what do the tools we have actually give us?

Let’s be honest about the current landscape.

GitHub Copilot is remarkable at what’s next — autocomplete, boilerplate, pattern matching. It understands your code in the moment. It has no memory of how you got there.

Sourcegraph is powerful for code search at scale — find every usage of a function across a monorepo. But it searches text, not intent. It can tell you where a pattern appears. It can’t tell you why it was introduced.

Notion AI / Confluence AI can summarize your documentation. But only if your documentation exists, is up to date, and actually captured the reasoning in the first place. For most projects, that’s a generous set of assumptions.

Git history is the closest thing we have to a reasoning trail — and its signal-to-noise ratio is brutal. “fix bug,” “wip,” “refactor v2,” “actually refactor v2.” If the decision was captured at all, it’s buried in a commit message written at 11pm under deadline pressure.

None of these tools are bad. They’re just solving a different problem. They help you work in a codebase. None of them help you understand one.

What Repository Intelligence would actually look like.

Imagine this instead.

You return to a project after two weeks. You open a query interface — think of it like a search bar, but for your codebase’s history and reasoning — and you type:

“Why did we switch from sequential to parallel processing in the data collection layer?”

And instead of getting a diff, you get back: the conversation that preceded the change, the constraint that made sequential processing a bottleneck, the alternative architecture that was considered and why it was ruled out, and the commit where it landed — with full context.

Or you ask: “What was the tradeoff we accepted when we chose SQLite over PostgreSQL here?” And the system surfaces the exact moment in the project’s history when that decision was made, why it made sense at the time, and what the known limitations were going in.

That’s Repository Intelligence. Not code search. Not documentation generation. Not autocomplete. A system that treats the evolution and reasoning of a codebase as a first-class queryable artifact — as important as the code itself.

It doesn’t fully exist yet.

There are early gestures toward it — tools that summarize PRs, tools that generate changelogs, tools that try to link issues to code. But none of them are built around the core premise: that why is as important as what, and that a codebase without its reasoning is only half a knowledge base.

The builders who feel this most acutely aren’t the ones working on massive enterprise codebases with dedicated documentation teams. It’s the solo builders, the small teams, the students and researchers moving fast across complex systems — the people for whom the cost of context decay is felt personally, every time they sit back down at their machine.

If you’ve ever written a note to future-yourself inside a codebase, you already know why this matters.

I’ve done it more times than I can count. Migration files as a breadcrumb trail. Handoff documents as memory prosthetics. Design docs written not for a team, but for the version of me who’ll open this folder three weeks from now and have no idea what I was thinking.

The question isn’t whether context decay is a real problem. Every builder reading this has already answered that question with their own workarounds.

The question is whether we keep doing it manually — or whether we finally build the tool that does it for us.
