On a Friday evening in February 2026, I clicked “Publish to Web” on a fully functional analytics dashboard and nothing happened. The dashboard worked. The SQL views were validated. The metrics matched. But Northeastern’s Power BI tenant had disabled public sharing for student accounts. There was no override. I had built a system that could not reach a single user.

This was not a notebook or a class demo. It was a query-driven analytics system with PostgreSQL on the backend, analytical SQL views for SLA compliance and resolution metrics, and a Power BI semantic model whose measures had been cross-checked against raw SQL output. I had spent weeks filtering Boston 311 data to the 2024 calendar year, cleaning 224,869 service requests, resolving type mismatches, and designing a warehouse structure that separated raw data from analytical logic.

Then I clicked “Publish to Web.”

Nothing happened.

I tried again. Checked the menus. Searched the documentation. And then I found the answer buried in a Microsoft support page: my Northeastern University Power BI account did not have permission to publish dashboards to the public internet. The feature was disabled at the tenant level. There was no workaround. No setting to toggle. No email I could send to make it happen before my deadline.

I had built an analytics system that worked perfectly on my machine, inside a virtual machine, connected to a local database, rendered in a tool that could not share its output with the outside world.

The dashboard nobody could see.

That moment taught me more about the distance between student work and production engineering than any lecture had. The project itself is not extraordinary. What mattered were the decisions that came after that failure, because those were the ones that turned a portfolio piece into something closer to a real system.

The Problem with How We Build Projects

Across coursework, hackathons, and portfolio discussions, I have noticed the same pattern: most student projects end at the notebook.

You find a dataset, clean it in pandas, build visualizations, maybe train a model, write a README, and push it to GitHub. That is a valid way to learn.

But it usually stops before the harder questions begin. How does this reach a user? How does it run without my laptop? What happens when the deployment environment behaves differently from development? What happens when the data is messier than expected?

Those are engineering questions. And they are the questions that make a project feel like a system rather than an assignment.

What Is Boston 311, and Why Does It Matter?

Boston 311 is the city’s non-emergency public service request system. Residents use it to report issues like potholes, broken streetlights, missed trash pickups, illegal parking, graffiti, and needle cleanup. The city publishes this data openly, with each request showing when it was opened and closed, what the issue was, which department handled it, where it happened, and whether the city met its service-level target for resolution.

On the surface, it looks like a standard municipal dataset. In practice, it is rich enough to support operational questions that matter: Which neighborhoods are falling behind on SLA compliance? Is backlog growing or shrinking? Are some issue types consistently slower to resolve? Which ZIP codes are underperforming relative to the city average?

Those are not notebook questions. They require aggregation logic, benchmark comparisons, time-series analysis, and a presentation layer usable by someone who does not write SQL.

I worked with a subset of Boston’s full 311 archive, which spans millions of records across multiple years. I filtered it to the 2024 calendar year, which gave me approximately 224,869 service requests. Real data. Real messiness. Real edge cases.

The First Pipeline: Azure Data Studio, a Virtual Machine, and Power BI

My initial architecture was straightforward. I extracted the raw CSV from Boston’s open data portal, cleaned it in Python using pandas, and loaded it into a local PostgreSQL instance through Azure Data Studio.

The cleaning was not trivial. Datetime formats were inconsistent, some ZIP code fields contained corrupted values merged with latitude, longitude, and source data, SLA status values used inconsistent casing, and null handling varied across columns.

Once the data was clean, I loaded it into PostgreSQL and built analytical views on top of the raw table. These views computed the key metrics: total requests per month, open versus closed counts, SLA-eligible cases, SLA compliance rates, and median resolution hours. I chose SQL views to keep the metric logic separate from the frontend.

I then connected Power BI through a Windows virtual machine and built a semantic model as the validation layer. I designed a star schema with a fact table for 311 cases and dimensions for date, ZIP code, and department, then defined DAX measures for each KPI and tested them against raw SQL output for the same slices of data.

The Power BI dashboard had two pages. The first showed city-wide performance for 2024: total request volume, open backlog, SLA compliance trend, and monthly request volume. The second allowed ZIP-level comparison, showing how a selected ZIP’s SLA trend compared against the city average, with a risk and demand classification table ranking ZIPs by priority.

It worked. The numbers were validated. The visualizations were clean. The model was sound.

And then I tried to put it on the internet.

The Wall

The problem was simple and absolute. Northeastern’s Power BI tenant does not allow Publish to Web for student accounts. This is a university-level administrative setting. There is no override. There is no request form. It is just a wall.

In coursework, the deliverable is often a screenshot, a PDF, or a demo on your own machine. Nobody asks whether a stranger can open your dashboard in a browser without your credentials.

Outside the classroom, that is the point. An analytics system that cannot be reached by its audience is not really a system. It is a file on a hard drive.

I had two choices. Accept the limitation and submit what I had or rebuild the delivery layer from scratch using a stack that could actually reach the internet.

I rebuilt.

The Migration: From Power BI to Streamlit, From Local to Cloud

The first decision was the database. A local PostgreSQL instance was fine for development, but it could not support a hosted application, so I migrated to Supabase, a managed PostgreSQL service with connection pooling and SSL support built in.

This was not a drag-and-drop move. I recreated the raw table schema, rebuilt the analytical SQL views, migrated the cleaned dataset, and then connected a new application layer.

For that layer, I chose Streamlit. It let me build and deploy a public-facing analytics app quickly without needing a separate frontend stack. Most importantly, it could actually ship.

The architecture became:

Boston 311 CSV → Python Cleaning → Supabase PostgreSQL → Analytical SQL Views → Streamlit Application → Streamlit Cloud Deployment

Power BI was not discarded. It became the validation layer rather than the delivery layer. The DAX measures served as the specification that the PostgreSQL views had to match, while the star schema design influenced how I structured the views and queries. The system users would actually interact with was now Streamlit, backed by Supabase and deployed to the cloud.

The Bugs That Only Appear in Production

If the Power BI wall was the first lesson in production engineering, the deployment bugs were the second.

The IPv6 Connection Failure

The first time I deployed to Streamlit Cloud and connected it to Supabase, the application crashed with a single error: Cannot assign requested address.

The issue never appeared in local development. Supabase’s direct database host resolved to an IPv6 address, but Streamlit Cloud’s runtime environment did not support that connection path.

There was no warning. You deploy, it fails, and then you debug.

The fix was to switch from the direct Supabase host to the connection pooler host:

aws-0-us-west-2.pooler.supabase.com

One line in the configuration. Most of an evening to diagnose.

The Corrupted ZIP Codes

During data cleaning, I discovered that some ZIP code values in the source dataset were not ZIP codes at all. They were fragments of concatenated strings where the ZIP had been merged with latitude, longitude, and source fields:

02132,42.29,-71.15,Constituent Call

This kind of corruption does not appear when you preview the first twenty rows. It hides in the tail of the distribution. You find it when your aggregations produce unexpected values, or when a ZIP code that should not exist shows up in a dropdown.

The fix was a SQL-level extraction and validation step:

UPDATE cases_analytics_2024

SET location_zipcode = split_part(source, ‘,’, 1)

WHERE ...

Every ZIP code was then validated against a five-digit regex pattern: ^[0-9]{5}$. Anything that did not pass was flagged and handled.

This is what real-world data looks like. Not the clean CSVs in a textbook. Not the preprocessed Kaggle datasets where someone else already handled the edge cases. It is messy in ways you do not anticipate, and the only way to find the mess is to build a system that breaks when the mess exists.

The Logging Inflation Bug

Streamlit has a reactive execution model: every time a user changes a filter, interacts with a widget, or even resizes the browser, the script reruns from top to bottom.

That creates a problem for logging. If a page-view call sits at the top of the script, it fires on every rerun. A user who changes filters a few times can generate multiple page views in a single visit, which inflates the analytics.

To fix this, I wrote a log_once() function that uses Streamlit’s session state to ensure a page view is logged only once per page and filter combination:

def log_once(page: str, zip_selected: str | None = None):
    key = f"logged::{page}::{zip_selected or 'null'}"
    if st.session_state.get(key):
        return
    st.session_state[key] = True
    log_page_view(page, zip_selected)

Combined with a UUID-based session ID created once per browser session, this gave me cleaner usage data: total views, unique sessions, page-level analytics, and ZIP selection tracking, without collecting PII or IP addresses.

It was a small fix, but an important one. It turned the app from a demo into something that could observe its own usage honestly.

What the Product Actually Does

The live application has two pages.

Find My Area lets you select a ZIP code and compare it against the city average. Six KPI cards show the ZIP’s total requests, open requests, SLA rate, median resolution, the delta between ZIP and city SLA for the most recent month, and the city-wide SLA rate for context.

City Overview shows Boston-wide 311 performance for 2024, including total requests, open requests, SLA compliance rate, median resolution time, monthly request volume, and SLA trend over time.

The important part is not just what the app displays. It is how the numbers are produced. Every metric on both pages traces back to a SQL view, and every view traces back to a validated measure definition from the Power BI semantic model. The application does not invent a number.

        |

        |

There are multiple layers here, and each has one job. That separation is what made the rebuild possible.

When the Supabase connection failed, I did not have to rewrite my metrics. When the ZIP code issue appeared, I did not have to redesign the frontend. When logging inflated, I changed one small part of the application layer and left the rest alone.

That is why separation of concerns matters. Not because it looks good on an architecture diagram, but because it makes a system survivable when something breaks.

What This Project Changed for Me

The most valuable part of this project was not building the first dashboard. It was discovering everything that broke once I tried to make it accessible to real users.

That was the gap.

The difference between “I analyzed a dataset” and “I built something usable” was not another chart, another notebook, or another README. It was deployment, debugging, migration, validation, and redesign under constraint.

If I were giving one piece of advice to another student, it would be this: build the thing, then try to ship it. That is where the real learning starts.

Conclusion

The dashboard that nobody could see now has a public URL. It serves real metrics from a real database to real users. It was not built in a single clean sprint. It was built by running into walls, diagnosing failures, changing plans, and rebuilding layers I thought were finished.

That is what production engineering looked like in this project. Not the architecture diagram at the beginning, but the messy process of making something work outside the controlled environment where I built it.

If you have hit a similar deployment wall in your own project, I would genuinely like to hear what broke and how you fixed it.
